## 3. ROOT CAUSE ANALYSIS

### 3.1 Pipeline: `sales_daily_etl`

- **Error:** `Connection timeout to source database`
- **Failure Type:** TIMEOUT_ERROR
- **Root Cause:** The pipeline was unable to establish a connection to its source database within the configured timeout window. Probable causes include: (a) the source database is experiencing elevated load or is temporarily unavailable, (b) a network configuration change or firewall rule modification has disrupted connectivity between the ETL compute environment and the database host, or (c) connection pool exhaustion on the database side. Preliminary investigation should focus on source database health metrics and network path validation.
- **Upstream Sources Affected:** S3 raw sales bucket, Salesforce CRM API
- **Downstream Cascade:** This failure directly blocks `finance_reporting_etl`, `executive_dashboard`, and `sales_forecast_model`. Because `finance_reporting_etl` is itself a dependency for three additional systems, the true blast radius of this single timeout extends to **six downstream systems**.

### 3.2 Pipeline: `customer_data_warehouse`

- **Error:** `Schema mismatch: column 'revenue' not found`
- **Failure Type:** SCHEMA_ERROR
- **Root Cause:** The pipeline expects a column named `revenue` in one of its upstream source schemas, but that column is absent at runtime. This is consistent with one of the following scenarios: (a) an upstream schema migration was deployed to the CRM database, `support_tickets_etl`, or `web_events_stream` without coordinated notification to the data engineering team, (b) a breaking DDL change (column rename, column drop) was applied to a source table, or (c) a source system API version change altered the payload structure. The absence of a pre-execution schema validation contract allowed this mismatch to propagate to runtime failure.
- **Upstream Sources Affected:** CRM database, support_tickets_etl, web_events_stream
- **Downstream Cascade:** Blocks `marketing_attribution`, `churn_prediction_model`, and `customer_360_report`.

### 3.3 Pipeline: `finance_reporting_etl`

- **Error:** `Insufficient permissions on S3 bucket`
- **Failure Type:** PERMISSION_ERROR
- **Root Cause:** The IAM role or service account used by the `finance_reporting_etl` pipeline no longer has sufficient read/write permissions on the target S3 finance bucket. This is consistent with: (a) a recent IAM policy change or rotation of access credentials, (b) an S3 bucket policy update that inadvertently revoked access for the pipeline's execution role, or (c) an AWS organizational Service Control Policy (SCP) change that restricted access. Additionally, even once this permissions issue is resolved, this pipeline remains blocked by the upstream failure of `sales_daily_etl`, creating a **dual-failure dependency**.
- **Upstream Sources Affected:** sales_daily_etl (currently failed), accounts_payable_db, S3 finance bucket
- **Downstream Cascade:** Blocks `board_reporting_dashboard`, `regulatory_compliance_export`, and `CFO_weekly_report`.

## 5. RECOMMENDED ACTIONS

The following remediation steps are ordered by **priority**, factoring in cascade dependencies and business criticality.

### Priority 1: Resolve `sales_daily_etl` (TIMEOUT_ERROR)
**Estimated Time:** 30 minutes | **Assigned to:** Data Platform On-Call

1. Verify source database health: check connection status, active sessions, CPU/memory utilization, and query queue depth.
2. Validate network connectivity from the ETL compute environment to the database host (ping, traceroute, security group rules, VPC peering status).
3. Check for recent infrastructure changes (firewall rules, DNS modifications, VPN tunnel status).
4. If the source database is healthy and reachable, increase the connection timeout threshold temporarily and retry the pipeline.
5. If the database is under excessive load, coordinate with the DBA team to identify and terminate blocking queries or scale resources.
6. Upon successful completion, confirm that downstream `finance_reporting_etl` input data is refreshed.

### Priority 2: Resolve `finance_reporting_etl` (PERMISSION_ERROR)
**Estimated Time:** 1 hour | **Assigned to:** Cloud Infrastructure / IAM Team

1. Identify the IAM role or service account associated with the `finance_reporting_etl` execution environment.
2. Audit the S3 finance bucket policy and the associated IAM policy for recent changes (use AWS CloudTrail to identify who modified the policy and when).
3. Restore the necessary `s3:GetObject`, `s3:PutObject`, and `s3:ListBucket` permissions for the pipeline's execution role.
4. Validate permissions by running a manual test read/write against the S3 finance bucket.
5. **Do not trigger a full pipeline retry until `sales_daily_etl` has completed successfully**, as this pipeline depends on its output.
6. Once both upstream data and permissions are confirmed, trigger a full pipeline rerun and validate output to `board_reporting_dashboard`, `regulatory_compliance_export`, and `CFO_weekly_report`.

### Priority 3: Resolve `customer_data_warehouse` (SCHEMA_ERROR)
**Estimated Time:** 1 hour | **Assigned to:** Data Engineering Team

1. Query each upstream source (CRM database, `support_tickets_etl` output, `web_events_stream`) to locate the expected `revenue` column.
2. Identify which source underwent a schema change: check recent DDL logs, migration history, and deployment records.
3. Determine whether the column was **renamed**, **dropped**, or **relocated** to a different table/view.
4. Implement the appropriate fix:
   - If renamed: update the pipeline's column mapping configuration to reference the new column name.
   - If dropped: coordinate with the source system owner to understand the intent and either restore the column or derive the `revenue` value from an alternative source.
   - If relocated: update the pipeline's source query to join or reference the new location.
5. Deploy the schema fix to the pipeline configuration and trigger a rerun.
6. Validate that `marketing_attribution`, `churn_prediction_model`, and `customer_360_report` receive complete and accurate data.
