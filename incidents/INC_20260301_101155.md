## 3. ROOT CAUSE ANALYSIS

### 3.1 Pipeline: `sales_daily_etl`
**Error:** Connection timeout to source database
**Failure Type:** TIMEOUT_ERROR

**Root Cause:** The pipeline was unable to establish a connection to its source database within the configured timeout threshold. Probable contributing factors include:

- **Network-level disruption** between the ETL compute environment and the source database host (e.g., VPC peering issue, security group modification, or transient network partition).
- **Source database overload or unavailability** — the source database may be experiencing high connection saturation, long-running queries, or an active maintenance window that was not communicated to the data engineering team.
- **Connection pool exhaustion** — if other processes or a retry storm consumed available database connections, this pipeline's connection request would be queued and ultimately timed out.

**Upstream Sources Affected:** S3 raw sales bucket, Salesforce CRM API
**Downstream Impact:** `finance_reporting_etl`, `executive_dashboard`, `sales_forecast_model`

---

### 3.2 Pipeline: `customer_data_warehouse`
**Error:** Schema mismatch — column `revenue` not found
**Failure Type:** SCHEMA_ERROR

**Root Cause:** The pipeline expects a column named `revenue` in one of its upstream source schemas, but this column is no longer present at the expected location. Probable contributing factors include:

- **Uncoordinated upstream schema change** — a team responsible for the CRM database, `support_tickets_etl`, or `web_events_stream` likely executed a schema migration (column rename, removal, or table restructure) without notifying downstream consumers or updating the data contract.
- **Absence of schema validation at ingestion** — the pipeline lacks a pre-execution schema validation check that would detect breaking changes before processing begins, resulting in a hard failure during transformation rather than a controlled, early-exit alert.
- **Missing data contract enforcement** — there is no formal, versioned data contract between the upstream source owners and this pipeline, allowing breaking changes to propagate silently.

**Upstream Sources Affected:** CRM database, `support_tickets_etl`, `web_events_stream`
**Downstream Impact:** `marketing_attribution`, `churn_prediction_model`, `customer_360_report`

---

### 3.3 Pipeline: `finance_reporting_etl`
**Error:** Insufficient permissions on S3 bucket
**Failure Type:** PERMISSION_ERROR

**Root Cause:** The IAM role or service account used by this pipeline no longer has adequate read/write permissions on the S3 finance bucket. Probable contributing factors include:

- **IAM policy modification or rotation** — a recent security review, policy tightening, or automated credential rotation may have revoked or altered the permissions required by this pipeline's execution role.
- **S3 bucket policy update** — the bucket policy on the S3 finance bucket may have been updated (e.g., restricting access to specific VPC endpoints or IP ranges) without accounting for the ETL service's access pattern.
- **Temporary credential expiration** — if the pipeline relies on assumed roles with temporary credentials (STS), a misconfiguration in the trust policy or session duration could result in expired or invalid credentials at execution time.

**Note:** This pipeline is *also* blocked by the upstream failure of `sales_daily_etl`. Even after the permission issue is resolved, this pipeline cannot fully succeed until `sales_daily_etl` is restored and backfilled.

**Upstream Sources Affected:** `sales_daily_etl`, `accounts_payable_db`, S3 finance bucket
**Downstream Impact:** `board_reporting_dashboard`, `regulatory_compliance_export`, `CFO_weekly_report`

## 5. RECOMMENDED ACTIONS

Actions are prioritized by **severity of business impact** and **dependency chain logic** — resolving upstream failures first to unblock the maximum number of downstream systems.

### Priority 1: Restore `sales_daily_etl` (Est. 30 min)
**Owner:** Data Infrastructure Team
**Actions:**
1. Immediately verify network connectivity between the ETL compute environment and the source database (ping, traceroute, VPC peering status, security group rules).
2. Check source database availability, active connection count, and whether any maintenance windows are in effect.
3. If the database is healthy, increase the connection timeout threshold temporarily and reattempt the pipeline.
4. If the database is overloaded, coordinate with the DBA team to terminate long-running queries or increase the connection pool limit.
5. Once connectivity is confirmed, trigger a manual backfill run for today's sales data.
6. Validate output data completeness before marking the pipeline as recovered.

**Rationale:** This pipeline sits at the top of the dependency chain; restoring it unblocks `finance_reporting_etl`, `executive_dashboard`, and `sales_forecast_model`.

### Priority 2: Resolve `finance_reporting_etl` Permission Error (Est. 1 hr, parallel with Priority 1)
**Owner:** Cloud Security / Platform Engineering Team
**Actions:**
1. Identify the IAM role and/or service account used by `finance_reporting_etl`.
2. Audit the IAM policy and S3 bucket policy change history (via CloudTrail) for modifications made in the last 72 hours.
3. Restore the required `s3:GetObject`, `s3:PutObject`, and `s3:ListBucket` permissions on the S3 finance bucket for the pipeline's execution role.
4. If a credential rotation occurred, issue new temporary credentials and update the pipeline's secret store reference.
5. Perform a dry-run permission test (`aws s3 ls` / `aws s3 cp --dryrun`) before triggering a full pipeline re-execution.
6. Once both this fix and `sales_daily_etl` restoration are complete, trigger a full backfill run of `finance_reporting_etl`.

**Rationale:** This pipeline has three high-value downstream consumers including regulatory compliance. The permission fix can proceed in parallel with Priority 1, but the pipeline cannot fully execute until `sales_daily_etl` is also restored.

### Priority 3: Resolve `customer_data_warehouse` Schema Mismatch (Est. 1 hr)
**Owner:** Data Engineering Team, in coordination with CRM / upstream source owners
**Actions:**
1. Query the upstream CRM database, `support_tickets_etl` output, and `web_events_stream` schema to determine which source previously provided the `revenue` column and what changed.
2. Check source system migration logs and deployment history for recent DDL changes.
3. If the column was renamed: update the pipeline's transformation logic to reference the new column name.
4. If the column was removed: coordinate with the upstream data owner to determine whether the column can be restored, or implement an alternative derivation logic.
5. Deploy the schema fix to the pipeline, run integration tests against a staging environment, and validate that the `revenue` field is correctly populated in the output.
6. Trigger a backfill run and validate downstream consumer compatibility (`marketing_attribution`, `churn_prediction_model`, `customer_360_report`).

**Rationale:** While business-critical, the downstream consumers of this pipeline (marketing, churn modeling) have slightly longer tolerance windows than the finance and regulatory systems addressed in Priorities 1 and 2.

### Post-Resolution Validation (All Pipelines)
1. Confirm all nine downstream systems have received fresh data and are operating normally.
2. Run data quality checks (row counts, null rates, value distributions) against the last known good state.
3. Send an all-clear communication to stakeholders once full recovery is confirmed.
