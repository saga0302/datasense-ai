## 3. ROOT CAUSE ANALYSIS

### 3.1 — `sales_daily_etl`: Connection Timeout to Source Database

**Root Cause:** The pipeline was unable to establish a connection to its source database within the configured timeout window. Probable contributing factors include: (a) elevated load or resource contention on the source database at the time of the scheduled ETL run, (b) network latency or transient network partition between the ETL compute environment and the database host, or (c) the source database connection pool being exhausted by competing processes. The pipeline's upstream sources — the S3 raw sales bucket and Salesforce CRM API — may also be contributing if the Salesforce API endpoint is experiencing degraded response times.

**Classification:** Infrastructure / Connectivity

**Compounding Factor:** Because `finance_reporting_etl` depends on output from `sales_daily_etl`, this timeout has a direct cascading effect, worsening the finance pipeline's failure condition even beyond its own independent permission error.

---

### 3.2 — `customer_data_warehouse`: Schema Mismatch (`revenue` Column Not Found)

**Root Cause:** The pipeline encountered a schema validation failure: the expected column `revenue` was not present in the incoming data payload. This strongly indicates that an upstream schema change was deployed to one of the source systems — most likely the CRM database — without corresponding updates to the `customer_data_warehouse` pipeline's ingestion logic, transformation mappings, or schema contract definitions. Common causes include: (a) an upstream team renamed, removed, or restructured the `revenue` column without notifying the data engineering team, (b) a CRM database migration or release was applied that altered the table schema, or (c) a schema evolution event in the `web_events_stream` or `support_tickets_etl` feed introduced an incompatible change.

**Classification:** Schema Governance / Change Management

**Key Concern:** Schema mismatches of this nature suggest a gap in schema contract enforcement and cross-team change communication protocols.

---

### 3.3 — `finance_reporting_etl`: Insufficient Permissions on S3 Bucket

**Root Cause:** The pipeline's IAM role or service account no longer has the required read/write permissions on the S3 finance bucket. This is most likely caused by: (a) a recent IAM policy change or S3 bucket policy update that inadvertently revoked or narrowed access for the pipeline's execution role, (b) an AWS credential or role rotation event that was not propagated to the pipeline's configuration, or (c) an organizational security policy enforcement (e.g., AWS SCP, bucket policy tightening) applied without coordination with the data engineering team.

**Classification:** Access Management / IAM Configuration

**Compounding Factor:** This pipeline is also a downstream dependent of `sales_daily_etl`. Even once the permission error is resolved, `finance_reporting_etl` cannot produce complete output until `sales_daily_etl` is also restored, creating a dual-dependency failure mode.

## 6. PREVENTION RECOMMENDATIONS

### Short-Term (Implement within 1–2 weeks)

| # | Recommendation | Purpose |
|---|---|---|
| 1 | **Implement automated schema contract testing** on all critical pipeline ingestion points. Use tools such as Great Expectations or dbt schema tests to enforce expected column presence, data types, and constraints before pipeline execution proceeds. | Prevents schema mismatch failures from reaching production. |
| 2 | **Establish an IAM change notification mechanism.** Configure AWS CloudTrail alerts for any IAM policy or S3 bucket policy modifications affecting data pipeline execution roles. Route alerts to the data engineering on-call channel. | Provides immediate visibility into permission changes that could break pipelines. |
| 3 | **Add connection health pre-checks and retry logic with exponential backoff** to `sales_daily_etl` and all database-dependent pipelines. Current timeout behavior fails hard with no recovery attempt. | Improves resilience to transient connectivity issues. |
| 4 | **Configure cascading dependency alerts.** When a parent pipeline fails, automatically flag all downstream dependents as "at risk" in the monitoring system and notify affected stakeholders proactively. | Reduces mean time to awareness for cascading failures. |

### Medium-Term (Implement within 1–2 months)

| # | Recommendation | Purpose |
|---|---|---|
| 5 | **Establish a formal schema change management process** requiring upstream teams to submit schema change requests through a review workflow before deploying changes that affect shared data contracts. Integrate with CI/CD pipelines to enforce. | Eliminates surprise schema changes that break downstream consumers. |
| 6 | **Implement infrastructure-as-code (IaC) governance** for all IAM policies and S3 bucket policies related to data pipelines. All changes must go through version-controlled pull requests with data engineering review. | Prevents unauthorized or uncoordinated permission changes. |
| 7 | **Build automated pipeline dependency mapping and impact analysis tooling** that can instantly determine the full blast radius of any pipeline failure and generate stakeholder-specific impact summaries. | Accelerates incident triage and improves communication during outages. |
| 8 | **Conduct a comprehensive resilience review** of all HIGH-severity pipelines. Identify single points of failure, implement redundant connection paths where feasible, and establish SLA-driven failover procedures for compliance-critical pipelines. | Reduces organizational risk exposure from data pipeline failures. |
