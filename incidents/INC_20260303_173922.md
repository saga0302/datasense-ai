## 3. ROOT CAUSE ANALYSIS

### 3.1 Pipeline: `sales_daily_etl` — TIMEOUT_ERROR

**Root Cause:** The pipeline was unable to establish a connection to the source database within the configured timeout window. Preliminary investigation points to one or more of the following contributing factors:

- **Network-level issue:** A connectivity disruption or latency spike between the ETL compute environment and the source database host, potentially caused by a security group change, VPC routing modification, or transient network congestion.
- **Source database overload:** The source database may be experiencing resource saturation (connection pool exhaustion, long-running queries, or elevated load), causing it to reject or delay new connections.
- **Configuration drift:** Connection string parameters, hostname resolution, or credential rotation may have introduced a mismatch that manifests as a timeout rather than an explicit authentication failure.

**Upstream Sources Affected:** S3 raw sales bucket, Salesforce CRM API.

**Cascade Note:** This failure directly blocks `finance_reporting_etl`, which depends on `sales_daily_etl` output. Even once `finance_reporting_etl`'s own permission issue is resolved, it cannot complete until `sales_daily_etl` delivers data.

---

### 3.2 Pipeline: `customer_data_warehouse` — SCHEMA_ERROR

**Root Cause:** The pipeline execution failed because it expected a column named `revenue` in an upstream source table or intermediate dataset, and that column was not present at runtime. The most probable causes are:

- **Uncoordinated upstream schema change:** A schema modification was made to one of the upstream sources (`CRM database`, `support_tickets_etl`, or `web_events_stream`) — such as a column rename, removal, or type change — without corresponding updates to the `customer_data_warehouse` pipeline's transformation logic or schema contract.
- **Source system deployment:** A CRM or application deployment may have altered the data model without notifying the data engineering team, bypassing our schema change communication process.
- **Missing schema validation gate:** The absence of a pre-execution schema validation step allowed the pipeline to proceed to transformation before detecting the incompatibility.

**Upstream Sources Affected:** CRM database, `support_tickets_etl`, `web_events_stream`.

---

### 3.3 Pipeline: `finance_reporting_etl` — PERMISSION_ERROR

**Root Cause:** The pipeline's execution role or service account lacks the required IAM/S3 permissions to read from or write to the designated S3 finance bucket. The most probable causes are:

- **IAM policy modification:** A recent change to IAM policies, bucket policies, or role trust relationships — potentially as part of a security hardening initiative or access review — inadvertently revoked or narrowed the permissions required by the pipeline's service account.
- **Credential or role rotation:** Scheduled or ad-hoc rotation of service account credentials or assumption roles may have introduced a mismatch between the pipeline's configured credentials and the currently valid permissions.
- **Bucket policy update:** An S3 bucket policy change (e.g., adding explicit deny statements, restricting to specific VPC endpoints, or enabling S3 Object Ownership controls) may have overridden the existing IAM grants.

**Upstream Sources Affected:** `sales_daily_etl` (also failed), `accounts_payable_db`, S3 finance bucket.

**Compound Impact:** This pipeline sits at the intersection of two failure vectors — its own permission issue *and* missing input from the failed `sales_daily_etl`. Both must be resolved for full recovery.

## 6. PREVENTION RECOMMENDATIONS

### Short-Term (Implement within 1–2 weeks)

| # | Recommendation | Rationale |
|---|---|---|
| 1 | **Implement pre-execution schema validation checks** on all warehouse and ETL pipelines. Pipelines should compare expected schemas against source schemas *before* beginning data processing and emit structured alerts on mismatch. | Would have caught the `customer_data_warehouse` failure before execution, providing earlier warning and a clearer diagnostic. |
| 2 | **Add automated IAM/S3 permission drift detection** using AWS Config rules or a policy-as-code tool (e.g., OPA, CloudFormation Guard). Alert the data platform team when permissions affecting pipeline service accounts are modified. | Would have flagged the `finance_reporting_etl` permission revocation immediately upon policy change, before the pipeline's next scheduled run. |
| 3 | **Implement connection health checks and proactive alerting** for all source database connections. A lightweight heartbeat query should run every 5 minutes with alerts if latency exceeds thresholds or connections fail. | Would have detected the `sales_daily_etl` connectivity issue before the pipeline's scheduled execution window. |

### Medium-Term (Implement within 1–2 months)

| # | Recommendation | Rationale |
|---|---|---|
| 4 | **Establish formal schema contracts** (e.g., using a schema registry or data contract framework) between upstream producers and downstream consumers. All schema changes must pass through a contract compatibility check before deployment. | Prevents uncoordinated schema changes from breaking downstream pipelines. Shifts failure detection left to the deployment stage. |
| 5 | **Implement automated dependency-aware pipeline restart orchestration.** When an upstream pipeline recovers, downstream dependents should automatically queue for re-execution in the correct topological order. | Reduces manual intervention during cascading failures and ensures recovery is complete across the full dependency graph. |
| 6 | **Conduct a comprehensive access review** of all pipeline service accounts and create a documented permissions baseline. Implement change approval workflows that require data platform sign-off before modifying IAM policies affecting pipeline roles. | Prevents inadvertent permission revocation and establishes accountability for access changes affecting production pipelines. |

### Long-Term (Implement within 1 quarter)

| # | Recommendation | Rationale |
|---|---|---|

| 8 | **Establish a formal data reliability SLA framework** with defined RPO/RTO targets for each critical pipeline, documented escalation paths, and regular game-day exercises to test incident response procedures. | Creates organizational accountability for data reliability and ensures the team can respond to multi-pipeline failures with a practiced, repeatable process. |
