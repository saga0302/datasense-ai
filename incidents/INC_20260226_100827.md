## 3. ROOT CAUSE ANALYSIS

### 3.1 — `sales_daily_etl` | TIMEOUT_ERROR

**Root Cause:** The pipeline was unable to establish a connection to the source database within the configured timeout threshold. Probable causes include:

- **Source database overload or unavailability:** The upstream source database (feeding from the S3 raw sales bucket and Salesforce CRM API) may be experiencing elevated load, network partitioning, or an unscheduled maintenance window.
- **Network-level disruption:** A transient or ongoing network issue between the pipeline's compute environment and the source database host could be preventing TCP handshake completion.
- **Connection pool exhaustion:** If concurrent processes or long-running queries have consumed all available database connections, new connection requests will queue and ultimately time out.

**Cascading Effect:** This pipeline is an upstream dependency for `finance_reporting_etl`, meaning the finance pipeline will remain blocked regardless of its own independent fix until `sales_daily_etl` is restored.

### 3.2 — `customer_data_warehouse` | SCHEMA_ERROR

**Root Cause:** The pipeline's transformation or load step expects a column named `revenue` in one of its upstream source schemas (likely the CRM database), but this column is absent at runtime. Probable causes include:

- **Uncoordinated upstream schema change:** A DDL alteration (column rename, removal, or restructuring) was applied to an upstream source table—most likely in the CRM database—without corresponding updates to the pipeline's transformation logic or schema contract.
- **Missing schema migration step:** A deployment or migration script that was intended to accompany the schema change may not have been executed against the data warehouse staging layer.
- **Environment drift:** The pipeline may be pointing to an environment (e.g., a restored backup or a development replica) where the `revenue` column was never provisioned.

### 3.3 — `finance_reporting_etl` | PERMISSION_ERROR

**Root Cause:** The pipeline's execution role or service account lacks the required IAM/S3 permissions to read from or write to the designated S3 finance bucket. Probable causes include:

- **IAM policy modification:** A recent change to IAM policies, bucket policies, or role trust relationships has inadvertently revoked the pipeline's access. This may have resulted from a security team policy rotation, infrastructure-as-code deployment, or manual console change.
- **Credential or role expiration:** The assumed role's session credentials or temporary access tokens may have expired without automatic renewal.
- **Bucket policy override:** An S3 bucket policy update (e.g., adding an explicit deny or restricting to specific VPC endpoints) may have been applied without accounting for the pipeline's access pattern.

**Compounding Factor:** Even after permission restoration, this pipeline will not produce valid output until `sales_daily_etl` (an upstream dependency) is also restored.

## 6. PREVENTION RECOMMENDATIONS

### Short-Term (Implement Within 1–2 Weeks)

| # | Recommendation | Rationale |
|---|---|---|
| 1 | **Implement schema contract testing** on all pipelines with external upstream dependencies. Deploy automated schema validation checks (e.g., using Great Expectations, dbt tests, or custom pre-flight checks) that run before transformation logic executes. | The `customer_data_warehouse` failure would have been caught before pipeline execution, providing a clear error and preventing silent downstream degradation. |
| 2 | **Add connection health pre-checks** to all database-dependent pipelines. Pipelines should validate source connectivity with a lightweight probe (e.g., `SELECT 1`) before initiating full extraction. | This enables faster detection, more precise alerting, and automatic retry logic before full pipeline timeout occurs. |
| 3 | **Implement IAM change monitoring** with automated alerts when policies affecting pipeline service accounts or data buckets are modified. Use CloudTrail event rules or AWS Config rules. | The S3 permission issue would have been flagged at the moment of policy change, not at pipeline execution time. |

### Medium-Term (Implement Within 1–2 Months)

| # | Recommendation | Rationale |
|---|---|---|
| 4 | **Establish a formal schema change management protocol** requiring upstream teams to file schema change requests through a centralized registry. All downstream pipeline owners must acknowledge and adapt before DDL changes are applied to production. | Eliminates the class of schema mismatch errors caused by uncoordinated upstream changes. |
| 5 | **Deploy pipeline dependency mapping and cascade alerting.** When an upstream pipeline fails, all downstream dependent pipeline owners and business stakeholders should be automatically notified with estimated impact scope. | Today's incident affected 9 downstream systems; proactive cascade alerting would have enabled parallel business-side contingency planning. |
| 6 | **Implement automated credential rotation with pipeline-aware scheduling.** Ensure that credential or role session renewals do not coincide with pipeline execution windows, and that rotation events trigger validation checks. | Prevents permission failures caused by credential expiration race conditions. |

### Long-Term (Implement Within Quarter)

| # | Recommendation | Rationale |
|---|---|---|
| 7 | **Adopt pipeline circuit breaker patterns** that detect repeated failures and gracefully degrade, serving last-known-good data to downstream consumers with staleness metadata rather than propagating hard failures. | Reduces blast radius of upstream failures on critical business reporting. |
| 8 | **Conduct quarterly data pipeline resilience reviews** examining failure modes, SLA adherence, and dependency chain fragility across all critical pipelines. | Ensures systemic reliability improvements are tracked and prioritized as the data platform scales. |
